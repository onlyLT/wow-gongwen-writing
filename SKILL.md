# 公文写作（政府口径）

把零散要点、领导意图、素材，起草成规范、得体、可上会/可发文的体制内公文。

## 适用与不适用

**适用**：法定公文 15 种 + 常见事务性材料，政府及职能部门口径（重政策落地、数据事实、务实平实）。

**不适用（先说明再决定）**：党委/纪检/组织/宣传的强政治口径专稿（本 skill 政府口径，可写但需你补党言党语）；纯法律文书、合同、判决书；对外新闻通稿/营销文案；已成稿的深度政治把关（涉密、重大决策表述须由本单位核稿人终审）。

## 文种速判表

| 用户意图 | 文种 |
|---|---|
| 向上级请求批准/指示（要回复） | **请示**（一文一事、单主送） |
| 向上级汇报工作/情况（不要回复） | **报告**（不得夹带请示事项） |
| 答复下级的请示 | **批复** |
| 布置工作、转发文件、周知事项给下级 | **通知** |
| 与不相隶属机关商洽/询问/请批 | **函**（对不相隶属机关用"请示"是硬伤） |
| 出台政策举措要求下级执行 | **意见**（实施/指导意见）或 **通知** |
| 表彰/批评/传达情况 | **通报** |
| 记载会议议定事项 | **纪要** |
| 会上讲话部署 | **领导讲话稿** |
| 报送做法成效（对上/对外亮点） | **政务信息/简报** 或 **汇报材料** |
| 定目标、分任务、排步骤 | **工作方案/实施方案** |

## 核心工作流（默认：先出大纲，再逐段填）

除非用户要"直接出成稿"或"套模板填空"，一律走以下步骤（第 6 步导出按需）：

1. **取料**：先通读用户已提供的一切材料（上级来文、旧稿、会议记录、要点清单），四项要素——①文种 ②发文机关/主送机关/行文方向（上行/下行/平行）③事由与核心诉求 ④关键事实与数据（时间、地点、数字、依据文件、落款）——能从材料提取的直接提取并复述确认，不再提问。若一次涉及多个文种（如"出一套材料"），先列文件清单、排定顺序，逐件走流程，不混收多件的料。只对**影响骨架的未知项**（文种、行文方向、事由诉求、主送机关）提问，合并为一轮；文种、方向这类封闭问题给出推荐判断和理由、以选择题形式问，不开放式反问。凡可用占位符解决的事实数据一律不问（一轮通常不超过 3～4 问，列出的问题超过 4 个即回头检查是否混入了可占位项）。仍缺的事实按下方「占位符约定」占位，不杜撰数字、文号、人名。
2. **定文种与行文方向**：先按上方「文种速判表」定文种，再读 `references/wenzhong-handbook.md` 对应条目核对结构骨架、行文方向与易错点（如"请示不得多头主送、一文一事、需批复"；"报告不得夹带请示事项"）。方向错是硬伤，先卡住。
3. **先出大纲/框架**：给出标题（发文机关+事由+文种）+ 分层骨架（每段一句话点明要写什么），**停下来等用户确认**。大纲骨架取自 `references/wenzhong-handbook.md` 对应文种的"结构骨架"。
4. **逐段填成稿**：确认后按骨架展开为完整正文，语言按 `references/style-and-diction.md`（政府口径文风、开头结尾套路、称谓用语）；标题拟写、谋篇分条、素材化用按 `references/craft-principles.md`（标题技法、分条法则、种子例句——仿写不粘贴）；并对照 `references/negative-checklist.md` 自查雷区。
5. **版式与规则校验**：按 `references/format-gbt9704.md` 核对版式要素（标题、主送、发文字号、成文日期、附件说明、抄送、印章位等）与《党政机关公文处理工作条例》行文规则，输出一份「校验清单」随稿附上。
6. **导出 Word（按需）**：用户要 .docx 时，把成稿组织成 JSON 规格喂给 `scripts/export_docx.py`，产出 GB/T 9704 近似版式的公文（红头/报头、发文字号、署名日期等；标题=方正小标宋二号，一级标题=黑体、二级=楷体、三级=仿宋加粗、正文=仿宋，均三号）。用法见下。导出后须在随稿提示中列出 docx **未包含**的要素（抄送、版记、页码、份号/密级、印章），注明由办公室套模板补全，防止用户误以为 Word 版要素齐全。

单文种成稿骨架见 `templates/`（可作为第 3 步大纲的加速起点）。

### 占位符约定（防杜撰，全流程通用）

- `【待补：xx】`——缺失的事实（数字、文号、人名、日期、落款等），须由用户补齐后方可定稿。
- `【示意·待核】`——为帮助用户体感而给出的参考量级/示例值，使用前必须核实替换。
- `【待核对原文】`——引用重要论述、上级文件表述拿不准时的标记，定稿前须对照原文核对。

除此三种外不用其他占位写法；宁可占位，绝不杜撰。

### 改稿/润色分支（用户拿已有稿来时走这条，不走取料-大纲）

按 `references/craft-principles.md`「改稿七步与优先级」执行三遍式：
1. **诊断**：七步顺查（文种→主送→逻辑→事实→语言→格式→错字），输出按 **P0（事实/数据/文种方向）→ P1（观点/结论/结构）→ P2（语言/格式/标点）** 分级的诊断报告，**停下等用户确认**改哪些。
2. **修改**：按确认范围逐级改（P0 先行），逐段润色先开头、再主体、后结尾标题。
3. **通读复核**：前后口径一致，数据/人名/地名/日期/引文二次确认，照常附 `negative-checklist.md` 校验清单。改动处逐条说明改了什么、为什么。

### 导出 Word 用法

```bash
python scripts/export_docx.py spec.json          # 按 JSON 里的 output 落盘
python scripts/export_docx.py spec.json -o x.docx # 覆盖输出路径
python scripts/export_docx.py --demo demo.json    # 生成示例 JSON 参考
```

- JSON 字段与正文块类型（`text`/`h1`/`h2`/`plain`、`red_header`/`brief_header`/`signer`/`wenhao`/`title`/`zhusong`/`signoff_*`/`fuzhu` 等）见 `scripts/export_docx.py` 顶部文档字符串——**编 JSON 前先读它**。
- 字体缺省即 GB/T 9704 标准层级（经实际发文核校）：标题=方正小标宋简体二号加粗、一级标题(h1)=黑体、二级标题(h2)=楷体_GB2312、三级标题(h3)=仿宋_GB2312 加粗、正文=仿宋_GB2312（标题外均三号）。单位另有要求时在 JSON 的 `fonts` 里覆盖；本机未装的字体 Word 会自动替换，不影响生成。
- 依赖 `python-docx`（`pip install -r scripts/requirements.txt`）。生成后 `【待补】` 占位仍保留，提示用户填定。
- 命令中的 `python` 若不可用或**静默失败**（退出码非 0 且无输出，典型如 Windows 商店占位 stub），改用 `uv run --with python-docx python`（推荐，无需预装依赖）或 `py -3`；脚本路径以本 skill 根目录为基准，从其他目录调用时写全路径。

**可移植性与降级（跨 agent 平台运行时）**：本 skill 的**核心写作能力零依赖**——第 1–5 步只产出 markdown/纯文本，任何 Agent Skills 兼容的 harness 都能跑。第 6 步导出 Word 是**可选能力**：
- 若运行环境**有** `python-docx` → 调 `export_docx.py` 出 .docx。
- 若**无法运行**（python-docx 未装、python 不可用或调用静默失败、无子进程权限）→ **降级**：照常交付 markdown/纯文本成稿 + 版式说明，并提示用户"本环境未装 python-docx，如需 Word 请 pip/uv 安装 python-docx 后重跑第 6 步，或把成稿贴入公文模板"。不因导出不可用而阻断主体交付。

## 参考索引（按需读，勿全量加载）

- `references/wenzhong-handbook.md` — 文种手册：法定 15 种 + 事务性材料，每种含 适用/行文方向/结构骨架/口径要点/易错点。**每次起草先读对应文种条目。**
- `references/format-gbt9704.md` — GB/T 9704-2012 版式要素、发文字号/标题/成文日期规则、上行下行平行文差异。
- `references/style-and-diction.md` — 政府口径文风、高频开头/过渡/结尾套路、称谓与规范用语、数字与计量书写。
- `references/negative-checklist.md` — 负面清单：政治性差错、格式硬伤、行文规则违规、文风套话过度、事实/数据风险，逐条自查。
- `references/craft-principles.md` — 写作内功：标题技法（六生成式）、谋篇骨架、口语→公文翻译法则、语言十弊、素材种子库、改稿七步与 P0/P1/P2 优先级。**成稿润色与改稿分支必读。**
- `scripts/export_docx.py` — 通用公文 → Word 导出器（数据驱动 JSON），任意文种复用。顶部文档字符串即 JSON 契约。

## 输出契约

- 走大纲流程时：先只给「标题 + 分层骨架」，明确请用户确认后再展开，不要一次冲到底。
- 出成稿时：给 ①完整正文（含版式位置提示）②校验清单（版式/行文规则/风险点逐条）③占位符汇总（`【待补】`/`【示意·待核】`/`【待核对原文】` 三类分别列出）。
- 涉密、重大决策、对外表述一律提示"须经本单位核稿人/保密审查终审"，不代替把关。
